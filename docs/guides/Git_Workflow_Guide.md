# 📌 Git 필수 작업 플로우 가이드

팀 프로젝트에서 코드를 안전하게 깃허브에 올리고 원격 서버에 업데이트하기 위한 **8단계 필수 프로세스**입니다.

---

## 🚀 8단계 작업 프로세스 및 핵심 이유

### 1. 본인 작업 중인 브랜치 명 확인
*   **명령어**: `git branch`
*   **왜 해야 할까요?** 내가 지금 어느 브랜치에 있는지(예: `main`, `develop`, `feat/teammates` 등) 확인하지 않고 무심코 코드를 푸시하면, 엉뚱한 브랜치의 코드를 덮어쓰거나 배포 사고가 발생할 수 있습니다. 길을 잃지 않기 위한 첫 번째 나침반입니다.

### 2. 작업 중인 것을 커밋 (또는 스테이징)
*   **명령어**: `git add .` (그리고 필요 시 임시 커밋 `git commit -m "작업 백업"`)
*   **왜 해야 할까요?** 현재 내가 수정한 파일들을 안전하게 `Git`의 관리 하에 묶어두는 과정입니다. 작업 내역을 묶어두지 않은 상태로 다른 브랜치의 코드를 가져오게(merge) 되면, 내가 작업 중이던 코드가 유실될 위험이 높습니다.

### 3. Fetch Origin (원격 저장소 상태 파악)
*   **명령어**: `git fetch origin`
*   **왜 해야 할까요?** 내가 코드를 짜고 있던 사이, 다른 팀원들이 `develop` (또는 `main`) 브랜치에 새로운 코드를 올려두었을 수 있습니다. `fetch`는 내 코드를 덮어쓰지 않으면서 "원격 저장소에 어떤 변화가 있었는지" 정보만 안전하게 내려받는 작업입니다.

### 4. Merge origin/develop (최신 코드 병합)
*   **명령어**: `git merge origin/develop`
*   **왜 해야 할까요?** 내 코드에 팀원들이 추가한 최신 버전의 코드를 합치는(Merge) 과정입니다. 프로젝트는 혼자 만드는 것이 아니므로, 내가 만든 새 기능이 다른 사람들의 최신 코드와도 잘 맞물려 돌아가는지 로컬에서 미리 합쳐보고 테스트해야 합니다.

### 5. 나의 코드와 충돌(Conflict) 확인 및 해결
*   **작업 내용**: VSCode 등에서 `Conflict` 발생 여부를 체크하고, `Accept Current/Incoming`을 통해 에러를 잡습니다.
*   **왜 해야 할까요?** 내가 수정한 파일(예: `kanban.tsx`)의 10번째 줄을 다른 팀원도 똑같이 수정했다면, Git은 누구의 코드를 선택해야 할지 몰라 충돌을 냅니다. 이를 푸시하기 전에 내 컴퓨터에서 직접 판단하여 코드가 고장나지 않도록 수정비해야 합니다.

### 6. 로컬에 있는 커밋 메시지 생성 (병합 및 최종 커밋)
*   **명령어**: `git commit -m "feat: 칸반 상태 추가 및 최신 develop 병합"`
*   **왜 해야 할까요?** 충돌을 해결했거나 최신 코드를 성공적으로 반영했다면, "이러이러한 기능 작업을 마쳤고 코드를 정상적으로 합쳤음"을 증명하는 고유한 명찰(커밋 메시지)을 달아주어야 합니다. 나중에 동료들이 코드 리뷰를 할 때 이 메시지를 보고 어떤 작업인지 한 번에 알 수 있습니다.

### 7. Push (원격 저장소 업로드)
*   **명령어**: `git push origin [현재브랜치명]`
*   **왜 해야 할까요?** 지금까지의 1~6번은 모두 '내 컴퓨터(로컬)' 디스크 안에서만 일어난 일입니다. 팀원들과 이를 공유하고 실제 서버에 반영하려면, 해결된 최종본을 GitHub 원격 저장소(`origin`)로 쏘아 올려야(Push) 합니다.

### 8. 다시 코드 작업 시작
*   **왜 해야 할까요?** 한 사이클의 작업이 안전하게 깃허브에 저장되었으므로 마음 편하게 다음 기능(티켓/태스크) 개발에 착수할 수 있습니다.
