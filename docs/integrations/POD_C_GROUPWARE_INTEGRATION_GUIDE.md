# Pod-C 그룹웨어 연동 가이드

## 1. 문서 목적
이 문서는 WorkPresso에서 Pod-C를 중심으로 Pod-A, Pod-B, Pod-D, Pod-E를 어떤 방식으로 연동해야 하는지와, 그룹웨어 성격에 맞는 실제 사용 방식을 정리한 운영 가이드입니다.

---

## 2. 역할 정의
- **Pod-A**: 결재 문서/정책/공식 문서의 원본 시스템
- **Pod-B**: 일정/업무 시간 관리의 원본 시스템
- **Pod-C**: AI 검색, 요약, 질의응답, 워크플로우 오케스트레이션 허브
- **Pod-D**: 회의 음성/STT/회의록의 원본 시스템
- **Pod-E**: 채널형 협업/공유/커뮤니케이션 시스템

### 핵심 원칙
1. **원본 데이터는 각 Pod가 보관**한다.
2. **Pod-C는 검색/이해/추천/실행 보조**를 담당한다.
3. **실제 생성/수정/승인 같은 쓰기 작업은 원본 Pod API를 호출**한다.
4. **권한, 상태, 공식성 판단은 원본 Pod 기준**으로 처리한다.
5. **Pod-C는 그룹웨어의 두뇌**, A/B/D/E는 **원장 시스템**으로 사용한다.

---

## 3. 왜 이렇게 써야 하는가
그룹웨어에서는 아래 3가지가 분리되어야 합니다.
- **공식 기록**: 문서, 결재, 일정, 회의록
- **협업 대화**: 채팅, 공유, 실시간 커뮤니케이션
- **AI 보조**: 요약, 검색, 질의응답, 액션 제안

Pod-C가 모든 것을 직접 저장하거나 상태를 바꾸기 시작하면,
- 승인 권한 우회 위험
- 원본/검색 데이터 불일치
- 회귀 테스트 복잡도 증가
- 책임 경계 불명확
문제가 생깁니다.

따라서 Pod-C는 **검색과 실행 오케스트레이션**에 집중해야 합니다.

---

## 4. Pod별 권장 연동 방식

### 4.1 Pod-A ↔ Pod-C
#### 연동 시점
- 문서가 **최종 승인(APPROVED)** 되었을 때
- 승인 문서가 재편집/재오픈 되었을 때

#### 연동 방식
- 승인 완료 문서만 Pod-C 지식 인덱스에 적재
- 문서 수정/재오픈 시 기존 인덱스 갱신 또는 stale 정리
- Pod-C 답변에는 문서 링크와 승인 상태를 같이 노출

#### 권장 사용 예시
- "재택근무 승인 기준 알려줘"
- "최신 보안 정책 문서 요약해줘"
- "팀 운영 정책 원문 링크 보여줘"

#### 그룹웨어 관점
- Pod-A는 **공식 근거 문서 저장소**
- Pod-C는 **승인된 문서를 찾아주는 비서**

---

### 4.2 Pod-B ↔ Pod-C
#### 연동 시점
- 일정 생성/수정/삭제 시
- 사용자가 자연어로 일정 등록을 요청할 때

#### 연동 방식
- Pod-B 일정 메타데이터를 Pod-C에 인덱싱
- Pod-C는 일정 생성 의도를 해석해 Pod-B API를 호출
- 시간/참석자 정보가 부족하면 Pod-C가 재질문

#### 권장 사용 예시
- "내일 오후 2시에 디자인 리뷰 일정 잡아줘"
- "이번 주 고객 미팅만 보여줘"
- "이번 달 마감 일정 중 내가 담당인 것만 정리해줘"

#### 그룹웨어 관점
- Pod-B는 **실제 일정 원장**
- Pod-C는 **일정을 만들고 찾게 도와주는 비서**

---

### 4.3 Pod-D ↔ Pod-C
#### 연동 시점
- STT 완료 시
- 회의록 정제 완료 시
- 액션 아이템 추출 완료 시

#### 연동 방식
- Pod-D의 transcript, summary, participants, action_items를 Pod-C에 인덱싱
- Pod-C는 회의 맥락 기반 질의응답 수행
- 필요 시 액션 아이템을 Pod-B 일정으로 전환

#### 권장 사용 예시
- "지난주 운영회의에서 장애 대응 담당 누구였지?"
- "어제 회의 액션아이템만 정리해줘"
- "회의록 기준 후속 작업 일정을 생성해줘"

#### 그룹웨어 관점
- Pod-D는 **집단 지식의 원문 저장소**
- Pod-C는 **회의 맥락을 기억하고 재활용하는 계층**

---

### 4.4 Pod-E ↔ Pod-C
#### 연동 시점
- 채널 대화에서 문서/일정이 공유될 때
- 긴 논의를 요약하거나 업무 객체로 전환할 때
- 대화 내용을 기반으로 문서/일정/회의 작업을 생성할 때

#### 연동 방식
- Pod-E는 실시간 대화와 공유의 중심
- Pod-C는 대화를 요약하고, 링크된 업무 객체를 해석하며, 다음 작업을 제안
- Pod-E 대화에서 필요 시 Pod-C를 호출해 Pod-A/B/D 생성 플로우로 연결

#### 권장 사용 예시
- "이 대화 3줄로 요약해줘"
- "액션아이템만 뽑아줘"
- "이 논의 기준으로 일정 만들어줘"
- "이 논의 내용으로 결재 문서 초안 만들어줘"

#### 그룹웨어 관점
- Pod-E는 **대화 공간**
- Pod-C는 **대화 → 실행 가능한 업무 객체 전환기**

---

## 5. 추천 연동 구조

### 5.1 Event 기반 연동
업무 객체가 확정되면 Pod-C에 자동 반영합니다.
- Pod-A 최종 승인 → Pod-C 인덱싱
- Pod-B 일정 생성/수정 → Pod-C 인덱싱
- Pod-D STT/정제 완료 → Pod-C 인덱싱
- Pod-E 중요 대화 요약 저장 → Pod-C 인덱싱(선택)

### 5.2 Tool Calling 기반 연동
사용자가 Pod-C에서 실행을 요청하면 원본 Pod API를 호출합니다.
- 일정 생성 → Pod-B 호출
- 결재 문서 초안 생성 → Pod-A 호출
- 회의 후속 액션 등록 → Pod-B 호출
- 채널 요약/공유 → Pod-E 호출

### 5.3 운영 원칙
- **읽기/검색은 Pod-C 중심**
- **쓰기/실행은 원본 Pod 중심**
- **권한 검증은 원본 Pod에서 최종 판단**

---

## 6. 그룹웨어에 맞는 대표 사용 시나리오

### 시나리오 A. 회의 후 실행 전환
1. Pod-D에서 회의 녹음 업로드
2. STT 및 회의록 정제 완료
3. Pod-C가 회의 요약과 액션 아이템 정리
4. 사용자가 "담당자 일정으로 등록해줘" 요청
5. Pod-C가 Pod-B 일정 생성 API 호출
6. 필요 시 Pod-A 결재 초안까지 생성

**효과**: 회의가 기록에서 끝나지 않고 실행으로 이어짐

---

### 시나리오 B. 승인 문서 기반 사내 질의응답
1. Pod-A에서 정책 문서 최종 승인
2. Pod-C에 자동 인덱싱
3. 사용자가 "출장비 기준 뭐야?" 질문
4. Pod-C가 승인 문서 기반으로 답변
5. 필요 시 Pod-E 채널에 해당 문서 링크 공유

**효과**: 정책/규정 탐색 비용 감소

---

### 시나리오 C. 채팅에서 업무 객체 생성
1. Pod-E 프로젝트 채널에서 논의 진행
2. 사용자가 "이 대화 기준으로 일정 만들어줘" 요청
3. Pod-C가 대화 요약 후 Pod-B 일정 등록
4. 사용자가 "결재 초안도 만들어줘" 요청
5. Pod-C가 Pod-A 초안 생성 보조

**효과**: 메신저 대화가 실제 업무 자산으로 전환됨

---

### 시나리오 D. 프로젝트 브리핑 자동화
사용자가 Pod-C에 "프로젝트 X 현재 상태 요약해줘" 요청하면,
- Pod-A: 최근 승인 문서
- Pod-B: 예정 일정
- Pod-D: 최근 회의 액션 아이템
- Pod-E: 최근 공유 링크/논의 맥락
을 종합해 브리핑을 생성합니다.

**효과**: 팀장/PM/리더용 상태 브리핑 자동화

---

## 7. Pod-C에 맡기면 좋은 것 / 안 좋은 것

### Pod-C에 맡기면 좋은 것
- 사내 지식 검색
- 회의/채팅/문서 요약
- 액션 아이템 추출
- 다음 행동 제안
- 여러 Pod 데이터 종합 브리핑
- 자연어 기반 일정/문서 생성 보조

### Pod-C에 맡기면 안 좋은 것
- 승인 상태 직접 변경
- 공식 문서 원본 저장소 대체
- 일정 원본 저장소 대체
- 권한 우회 처리
- 미승인 문서를 공식 근거처럼 답변

---

## 8. 운영 리스크 및 예외 처리
- **미승인 문서 노출 위험**: Pod-A 상태 필터를 강제해야 함
- **중복 인덱싱 위험**: `source_type + source_id` 기준 idempotent upsert 필요
- **원본 변경 후 구 지식 잔존 위험**: 수정/재오픈 시 stale vector 정리 정책 필요
- **툴 호출 오작동 위험**: Pod-C가 확신이 낮으면 바로 생성하지 말고 재질문해야 함
- **권한 불일치 위험**: Pod-C가 보여주는 요약도 원본 Pod 권한 기준으로 제한해야 함

---

## 9. 권장 우선순위
1. **Pod-D → Pod-C**: 회의록 인덱싱 고도화
2. **Pod-B ↔ Pod-C**: 일정 생성 Tool Calling 계약 확정
3. **Pod-A → Pod-C**: 승인 문서 기반 답변 품질 고도화
4. **Pod-E ↔ Pod-C**: 대화 요약과 업무 객체 전환 UX 확장

---

## 10. 결론
WorkPresso에서 가장 그룹웨어다운 구조는,
- **Pod-A/B/D/E를 원본 시스템으로 유지**하고
- **Pod-C를 회사 업무의 두뇌와 오케스트레이터로 사용하는 방식**입니다.

이 구조를 따르면 검색 품질, 업무 자동화, 협업 생산성을 모두 높이면서도,
공식 기록과 권한 체계를 무너뜨리지 않을 수 있습니다.
