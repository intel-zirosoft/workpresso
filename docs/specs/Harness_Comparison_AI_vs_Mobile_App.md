# 하네스 개념 비교 정리
## AI 기반 개발 하네스 vs WorkPresso 모바일 앱 하네스

## 문서 목적
본 문서는 일반적으로 알려진 **AI 기반 개발 하네스**와, WorkPresso 모바일 앱에서 도입한 **모바일 앱 하네스**의 차이를 명확히 설명하기 위한 비교 문서입니다.

---

# 1. 먼저 결론
같은 "하네스"라는 용어를 쓰지만, 두 개는 목적과 대상이 다릅니다.

- **AI 기반 개발 하네스**는 AI가 만든 결과물을 검증하기 위한 프레임워크입니다.
- **WorkPresso 모바일 앱 하네스**는 기존 웹 서비스를 모바일 앱 안에서 실행하고 제어하기 위한 런타임 구조입니다.

즉,

> AI 개발 하네스는 **AI 산출물 검증용 하네스**이고,  
> WorkPresso 앱 하네스는 **웹 기능 모바일 실행용 하네스**입니다.

---

# 2. AI 기반 개발 하네스란 무엇인가
질문에서 말씀하신 하네스는 일반적으로 아래 흐름을 의미합니다.

## 2-1. 전형적인 구조
1. 입력(Prompt / Task)
2. AI 코드 생성
3. 샌드박스 실행 환경
4. 자동 검증
   - 유닛 테스트
   - 린트
   - 타입 체크
   - 보안 검사
5. 결과 판단
   - 통과 시 채택
   - 실패 시 AI 재호출

## 2-2. 핵심 목적
이 구조의 핵심 목적은 다음과 같습니다.

- AI가 만든 코드를 바로 신뢰하지 않고
- 격리된 환경에서 실행하고
- 테스트와 검사로 검증한 뒤
- 최종 채택 여부를 결정하는 것

즉, 이 하네스는
**AI가 생성한 결과물을 통제하고 검증하기 위한 자동화 프레임워크**입니다.

---

# 3. WorkPresso 모바일 앱 하네스란 무엇인가
WorkPresso에서 말하는 하네스는 AI 코드 생성 검증과는 전혀 다른 개념입니다.

이 프로젝트의 모바일 하네스는,
기존 `apps/web` 중심의 웹 기능을 모바일 앱 안에서 재사용하기 위해 만든 **실행 프레임워크**입니다.

## 3-1. 전형적인 구조
1. 사용자가 모바일 앱 탭 진입
2. Native App Shell 진입
3. WebView가 기존 웹 URL 로드
4. 앱이 런타임 제어 수행
   - 로딩 상태 관리
   - 에러/재시도
   - 뒤로가기 처리
   - 외부 링크 분기
   - 내부 링크 라우팅
   - 딥링크 연결
5. 필요 시 웹 ↔ 네이티브 브리지 통신

## 3-2. 핵심 목적
이 구조의 핵심 목적은 다음과 같습니다.

- 기존 웹 기능을 최대한 재사용하고
- 앱다운 UX는 네이티브 셸에서 제공하고
- 필요할 때 웹과 네이티브를 연결하는 것

즉, 이 하네스는
**웹 서비스를 모바일 앱 안에서 안정적으로 실행하고 제어하기 위한 구조**입니다.

---

# 4. 두 하네스의 차이

| 항목 | AI 기반 개발 하네스 | WorkPresso 모바일 앱 하네스 |
|---|---|---|
| 목적 | AI가 생성한 코드 검증 | 웹 기능을 모바일에서 실행/제어 |
| 대상 | AI 산출물(코드, 패치, 변경 결과) | 기존 웹 화면, 웹 라우트, 웹 기능 |
| 입력 | Prompt, Task, 요구사항 | 사용자 탭 클릭, 웹 URL, 딥링크, 브리지 이벤트 |
| 실행 환경 | 샌드박스, 테스트 러너, 정적 분석기 | Expo 앱 셸, WebView, 브리지, 딥링크 라우터 |
| 핵심 처리 | 생성 → 실행 → 검증 → 재생성 | 로드 → 표시 → 제어 → 네이티브 연동 |
| 검증 기준 | 테스트/린트/타입체크/보안 | 로딩 안정성, 세션, 뒤로가기, 외부 링크, 내부 라우팅 |
| 실패 시 대응 | AI 재호출 또는 코드 수정 | 재시도, 라우팅 보정, 브리지 처리, UX 보완 |
| 결과물 | 채택 가능한 코드 | 실행 가능한 모바일 사용자 경험 |

---

# 5. WorkPresso 앱 하네스는 실제로 무엇을 하느냐
WorkPresso 모바일 앱 하네스는 다음 역할을 합니다.

## 5-1. App Shell 제공
- Expo / React Native 기반 앱 구조
- 탭 네비게이션 구성
- Safe Area 및 기본 모바일 UI 제공

## 5-2. Web Harness 제공
- 각 탭에서 대응하는 웹 경로를 WebView로 로드
- 예: 문서, 일정, 채터, 도우미, 음성 기능 재사용

## 5-3. Runtime Control 수행
- 로딩 상태 표시
- 에러 감지 및 재시도
- 현재 URL 추적
- 안드로이드 뒤로가기 처리
- 외부 링크 분기
- 내부 링크의 앱 탭 연결

## 5-4. Bridge 제공
- 웹이 직접 할 수 없는 앱 기능을 Native Layer가 대신 처리
- 현재 기준 예시:
  - 외부 URL 열기
  - 기기 정보 조회
  - 세션 상태 감지
  - 라우트 변경 감지
  - 웹 런타임 오류 감지

즉 WorkPresso 앱 하네스는
**기존 웹 기능을 모바일 앱 안에서 사용자 친화적으로 실행할 수 있도록 감싸는 런타임 계층**입니다.

---

# 6. 왜 같은 하네스라는 단어를 쓰는가
두 개는 목적은 다르지만 공통점이 하나 있습니다.

둘 다 어떤 시스템을 그냥 직접 다루지 않고,
**바깥에서 감싸고 통제 가능하게 만드는 틀**이라는 점입니다.

- AI 개발 하네스: AI가 만든 코드를 감싸서 검증
- 모바일 앱 하네스: 기존 웹 기능을 감싸서 모바일에서 실행

즉 하네스라는 단어의 공통 의미는,

> “복잡한 시스템을 바로 쓰지 않고, 안전하고 통제 가능한 환경으로 감싸는 구조”

라고 볼 수 있습니다.

---

# 7. 발표용 설명 문구
## 7-1. 짧은 설명
제가 처음 생각했던 하네스는 AI가 만든 코드를 테스트하고 검증하는 자동화 프레임워크에 가까웠습니다. 반면 이번 WorkPresso 모바일 앱에서 말하는 하네스는, 기존 웹 서비스를 모바일 앱 안에서 실행하고 제어하기 위한 런타임 구조입니다. 즉 하나는 AI 산출물 검증용이고, 다른 하나는 웹 기능 모바일 실행용입니다.

## 7-2. 1분 발표용 설명
일반적으로 AI 개발에서 하네스라고 하면, AI가 생성한 코드를 샌드박스에서 실행하고 테스트와 린트, 타입 체크를 거쳐 통과 여부를 판단하는 검증 프레임워크를 의미합니다. 그런데 WorkPresso 모바일 앱에서의 하네스는 전혀 다른 의미입니다. 이 하네스는 기존 웹 기능을 모바일 앱 안에서 재사용하기 위해, Expo 기반 네이티브 앱 셸과 WebView, 브리지, 딥링크 처리 구조를 묶어 놓은 실행 프레임워크입니다. 즉 AI 개발 하네스가 생성 결과를 검증하는 구조라면, WorkPresso 앱 하네스는 웹 기능을 모바일 환경에서 안정적으로 실행시키는 구조라고 설명할 수 있습니다.

---

# 8. 한 줄 정리
**AI 개발 하네스는 생성된 코드를 검증하는 구조이고, WorkPresso 모바일 앱 하네스는 기존 웹 기능을 모바일 앱 안에서 실행하고 제어하는 구조입니다.**
