# WorkPresso 모바일 앱 MVP 시연 체크리스트

본 문서는 `docs/specs/Mobile_App_Conversion_MVP_Deadline_Plan.md` 기준으로  
**내일 안드로이드 Expo Go 시연을 바로 진행할 수 있도록** 실행 방법, 최종 점검 항목, 시연 순서를 한 번에 정리한 문서입니다.

> 기준 플랫폼: **안드로이드 / Expo Go**

---

## 1. 현재 시연 범위 요약

현재 `apps/mobile` MVP에서 시연 가능한 핵심 범위:

- Expo Go에서 앱 실행
- 하단 탭 진입
  - 홈
  - 채터
  - 일정
  - 문서
  - 도우미
  - 음성
- WebView 기반 화면 로딩
- 로딩 / 에러 / 재시도 UX
- 외부 링크는 외부 브라우저로 열기
- 내부 링크는 WebView 내부 유지
- Android 뒤로가기
- 현재 화면 새로고침
- 최소 브리지 확인
  - `window.WorkPressoMobile.getDeviceInfo()`
  - `window.WorkPressoMobile.openExternalUrl(url)`

이번 시연에서 제외:

- iOS 검증
- 네이티브 로그인
- 푸시 정식 연동
- 대규모 네이티브 화면 재작성

## 2. 시연 전 준비

### 2.1 모바일 환경 변수

`apps/mobile/.env.local`

```env
EXPO_PUBLIC_WEB_BASE_URL=http://<PC_LAN_IP>:3000
EXPO_PUBLIC_DEEP_LINK_SCHEME=workpresso
```

예:

```env
EXPO_PUBLIC_WEB_BASE_URL=http://172.30.1.23:3000
EXPO_PUBLIC_DEEP_LINK_SCHEME=workpresso
```

중요:

- `localhost` 대신 **PC LAN IP** 사용
- 모바일과 PC는 같은 Wi-Fi

### 2.2 웹 실행

```bash
cd apps/web
npm run dev -- --hostname 0.0.0.0 --port 3000
```

확인:

- [ ] 웹 서버가 켜져 있다.
- [ ] `http://<PC_LAN_IP>:3000`으로 모바일에서 접근 가능하다.

### 2.3 앱 실행

```bash
cd apps/mobile
npx expo start --host lan
```

확인:

- [ ] Expo Go에서 QR 스캔으로 앱 진입 가능하다.
- [ ] 첫 화면에서 흰 화면/즉시 크래시가 없다.

## 3. 내일까지 꼭 확인해야 할 Must Have

### A. 앱 셸 / 탭
- [ ] Expo Go에서 앱이 실행된다.
- [ ] 홈 탭이 열린다.
- [ ] 채터 탭이 열린다.
- [ ] 일정 탭이 열린다.
- [ ] 문서 탭이 열린다.
- [ ] 도우미 탭이 열린다.
- [ ] 음성 탭이 열린다.

### B. Web Harness 안정성
- [ ] 로딩 오버레이가 표시된다.
- [ ] 에러 발생 시 재시도 버튼이 보인다.
- [ ] 재시도 후 복구 가능하다.
- [ ] 내부 링크는 WebView 안에서 유지된다.
- [ ] 외부 링크는 외부 브라우저로 열린다.

### C. 모바일 기본 UX
- [ ] 상단 헤더가 상태바와 겹치지 않는다.
- [ ] `뒤로` 버튼이 WebView history 기준으로 동작한다.
- [ ] Android 하드웨어 뒤로가기가 WebView 내부 히스토리에서 동작한다.
- [ ] `새로고침` 버튼이 현재 화면 기준으로 동작한다.

### D. 최소 브리지
- [ ] 브리지 주입 로그(`BRIDGE_READY`)가 보인다.
- [ ] `getDeviceInfo()` 호출 결과가 보인다.
- [ ] `openExternalUrl(url)` 호출 후 외부 브라우저가 열린다.
- [ ] 외부 링크 호출 결과 로그(`EXTERNAL_URL_RESULT`)가 보인다.

## 4. 시연 직전 최종 점검

### 4.1 앱 기본 상태
- [ ] 하단 탭 텍스트가 정상 노출된다.
- [ ] 깨진 아이콘(X 박스)이 없다.
- [ ] 헤더 버튼이 상태바와 겹치지 않는다.

### 4.2 WebView 상태
- [ ] 첫 진입 시 로딩 메시지가 보인다.
- [ ] 로딩 종료 후 화면이 정상 표시된다.
- [ ] 서버 주소를 잘못 넣었을 때 에러/재시도 흐름이 문서와 동일하다.

### 4.3 링크 처리
- [ ] 내부 경로 이동 후 뒤로가기가 자연스럽다.
- [ ] 외부 링크 클릭 시 외부 브라우저 전환이 된다.
- [ ] 앱으로 복귀해도 현재 탭 흐름이 크게 깨지지 않는다.

### 4.4 브리지 검증 패널
- [ ] 개발 모드에서 상단에 `브리지 확인` 패널이 보인다.
- [ ] `기기정보` 버튼 탭 시 로그가 쌓인다.
- [ ] `외부링크` 버튼 탭 시 브라우저가 열리고 결과 로그가 남는다.

## 5. 실제 시연 순서

아래 순서대로 진행하면 됩니다.

### Step 1. 앱 실행
1. Expo Go로 QR 스캔
2. 앱이 열리면 홈 탭 첫 화면 확인
3. 상단 헤더와 하단 탭이 안정적으로 보이는지 확인

시연 포인트:

- “Expo Go에서 바로 실행되는 모바일 앱 MVP”
- “하단 탭 기반으로 주요 기능 접근 가능”

### Step 2. 탭 이동
순서:

1. 홈
2. 채터
3. 일정
4. 문서
5. 도우미
6. 음성

시연 포인트:

- 각 탭이 WebView 기반으로 정상 진입
- 탭 순서가 실제 업무 흐름에 맞게 정리됨

### Step 3. 외부 링크 처리
1. 채터 또는 문서 화면에서 외부 링크가 있는 위치로 이동
2. 링크 탭
3. 외부 브라우저 전환 확인
4. 다시 앱으로 복귀

시연 포인트:

- 외부 링크는 앱 내부에 갇히지 않음
- Web Harness 앱으로서 최소 UX 확보

### Step 4. 뒤로가기 / 새로고침
1. WebView 안에서 내부 페이지 몇 단계 이동
2. Android 하드웨어 뒤로가기 또는 상단 `뒤로` 버튼 사용
3. 현재 화면에서 `새로고침` 버튼 사용

시연 포인트:

- 모바일에서 꼭 필요한 최소 조작 지원
- “탭 이동 + 뒤로가기 + 새로고침”이 자연스럽게 동작

### Step 5. 브리지 확인
1. 상단 `브리지 확인` 패널에서 `기기정보` 탭
2. 로그에 `GET_DEVICE_INFO` → `DEVICE_INFO_RESULT` 확인
3. `외부링크` 탭
4. 브라우저 전환 후 앱 복귀
5. 로그에 `OPEN_EXTERNAL_URL` → `EXTERNAL_URL_RESULT` 확인

시연 포인트:

- 웹에서 앱 기능을 호출할 최소 경로가 이미 동작
- 이후 브리지 확장 기반 확보

## 6. 테스트 항목별 확인 기준

### 6.1 Expo Go 탭 확인
- [ ] 홈/채터/일정/문서/도우미/음성 탭이 모두 열린다.

### 6.2 WebView 로딩/에러/재시도
- [ ] 로딩 오버레이가 보인다.
- [ ] 네트워크 실패 시 에러 문구가 보인다.
- [ ] `다시 시도`로 복구 가능하다.

### 6.3 외부 링크 / Android 뒤로가기 / 새로고침
- [ ] 외부 링크 처리 흐름이 시연 순서에 반영되어 있다.
- [ ] Android 뒤로가기 시연 순서가 문서에 반영되어 있다.
- [ ] 새로고침 시연 순서가 문서에 반영되어 있다.

### 6.4 최소 브리지 확인
- [ ] `getDeviceInfo` 확인 절차가 문서에 포함되어 있다.
- [ ] `openExternalUrl` 확인 절차가 문서에 포함되어 있다.
- [ ] 브리지 확인 패널 사용법이 문서에 포함되어 있다.

## 7. 실패 시 우선 점검

### 네트워크
- [ ] `EXPO_PUBLIC_WEB_BASE_URL`이 PC LAN IP인지 확인
- [ ] PC와 안드로이드 기기가 같은 네트워크인지 확인
- [ ] 웹 서버가 `0.0.0.0` 바인딩인지 확인

### Expo
- [ ] `npx expo start -c --host lan`으로 캐시 삭제 후 재실행
- [ ] Expo Go를 완전히 종료 후 재진입

### WebView
- [ ] 잘못된 URL로 설정되지 않았는지 확인
- [ ] 로컬 웹 서버가 실제로 켜져 있는지 확인

### 브리지
- [ ] 개발 모드에서 `브리지 확인` 패널이 보이는지 확인
- [ ] `BRIDGE_READY` 로그가 먼저 찍히는지 확인
- [ ] `DEVICE_INFO_RESULT` 또는 `EXTERNAL_URL_RESULT`가 오는지 확인
