# WorkPresso 모바일 앱 전환 단기 실행 계획서 (D-1 MVP)

본 문서는 장기적인 앱 전환 전략이 아니라, **내일까지 앱 전환을 마쳐야 하는 상황**을 전제로 WorkPresso를 모바일 앱 형태로 최소 기능까지 완성하기 위한 **단기 실행 계획서**입니다.

---

## 1. 목표

### 1.1 최종 목표
내일까지 다음 조건을 만족하는 **동작 가능한 모바일 앱 MVP**를 확보한다.

- Expo Go에서 앱이 실행된다.
- 주요 업무 화면이 모바일 앱 탭에서 열린다.
- 사용자가 앱 안에서 기본적인 그룹웨어 확인/이동/처리를 할 수 있다.
- 앱이 웹의 단순 복제 수준을 넘어서 최소한의 모바일 UX를 가진다.

### 1.2 이번 범위의 핵심 정의
이번 목표는 **완전한 네이티브 앱 출시**가 아니라,
**“Web Harness 기반의 사용 가능한 그룹웨어 앱 MVP 확보”** 이다.

---

## 2. 범위 원칙

### 2.0 웹과 앱의 역할 차이
이번 MVP는 웹 화면을 그대로 모두 옮기는 것이 목표가 아니다.  
앱은 그룹웨어 특성상 **“이동 중 빠르게 확인하고 바로 처리하는 도구”** 로 정의한다.

따라서 앱에서는 아래 원칙을 우선 적용한다.

- 웹의 복잡한 작성/설정/관리 기능은 최대한 뒤로 미룬다.
- 앱에서는 **내 승인 대기 / 오늘 일정 / 읽지 않은 채터 / 빠른 질의** 같은 즉시 처리형 흐름을 우선 노출한다.
- 같은 기능이라도 웹과 앱에서 **노출 순서와 CTA를 다르게** 가져간다.
- 앱 탭과 메뉴는 “전체 기능 목록”보다 **지금 자주 쓰는 핵심 업무** 중심으로 단순화한다.

### 2.1 이번에 반드시 포함할 것
- 앱 셸(Expo Router)
- 탭 기반 진입 구조
- WebView 기반 핵심 화면 연결
- 외부 링크 처리
- 기본 로딩/에러/재시도 UX
- 모바일에서 꼭 필요한 최소 상호작용
  - 뒤로가기
  - 새로고침
  - 탭 이동

### 2.2 이번에 제외할 것
- 네이티브 인증 전환
- 푸시 알림 정식 연동
- 오프라인 저장
- 대규모 네이티브 화면 재작성
- 고급 브리지 기능 전체 구현
- 관리자/설정 영역 고도화
- 전체 UI 리디자인

### 2.3 이번 릴리즈의 판단 기준
이번 작업은 **기능 완전성보다 “실제 사용 가능성”** 을 우선한다.

## 3. 그룹웨어 특성을 고려한 MVP 우선순위

모바일 그룹웨어는 데스크톱 웹처럼 “모든 기능을 다 하는 도구”가 아니라,
**지금 해야 할 업무를 빠르게 확인하고 처리하는 도구** 여야 한다.

### 3.1 앱에서 우선 보여줘야 하는 것
1. 오늘 확인할 것
2. 즉시 처리할 것
3. 자주 열게 되는 것
4. 이동 중에도 처리 가능한 것

### 3.2 이번 MVP에서 우선할 기능
#### 1순위: 채터(Pod-E)
- 모바일에서 가장 자주 열 가능성이 높음
- 빠른 확인/답장/브리핑 확인에 적합

#### 2순위: 일정(Pod-B)
- 오늘 일정 확인 가치가 높음
- 회의 중심 그룹웨어에서 핵심 진입점

#### 3순위: 문서(Pod-A)
- 승인 대기/상태 확인 용도로 중요
- 다만 긴 편집보다 조회/승인이 우선

#### 4순위: 업무 도우미(Pod-C)
- 빠른 질의/검색/브리핑 확인에 적합
- 앱에서 비서형 진입점 역할 가능

#### 5순위: 음성(Pod-D)
- 회의/메모 관점에서 가치 높음
- 하지만 내일까지는 네이티브 녹음보다 우선 “화면 진입 가능” 수준 확보가 현실적

### 3.3 웹과 차별화해야 하는 핵심 영역

#### 홈
- 웹 홈을 그대로 보여주는 것보다 **오늘 처리할 일 요약** 성격이 더 중요함
- 앱에서는 “오늘 브리핑 / 빠른 이동 / 즉시 처리할 일” 구조를 우선 검토

#### 문서
- 웹: 작성, 편집, 결재선 설정, 상세 관리
- 앱: **승인 대기 문서 확인 / 상세 검토 / 승인 / 반려**
- 앱에서는 새 문서 작성 CTA를 기본 노출하지 않음

#### 일정
- 웹: 전체 캘린더 관리, 상세 편집
- 앱: **오늘 일정 / 이번 주 일정 / 회의 확인**
- 앱에서는 복잡한 편집보다 빠른 확인과 이동이 우선

#### 채터
- 웹: 전체 채널 구조 확인
- 앱: **읽지 않은 대화 / 브리핑 / 빠른 확인**
- 앱 메뉴와 우선순위에서 가장 앞쪽에 배치할 가치가 큼

#### 도우미
- 웹: 탐색형 질의도 가능
- 앱: **즉시 답이 필요한 업무 질문** 중심

#### 음성
- 웹: 기능 접근 중심
- 앱: 회의 직후 확인/기록 진입점 역할

### 3.4 앱 메뉴/탭 설계 원칙
- 앱 좌상단 메뉴와 탭은 웹 사이드바 축소판이 아니어야 함
- 앱 메뉴는 **프로필 + 핵심 기능 카드형** 구조를 우선 적용
- 메인 메뉴 후보는 문서 / 채터 / 일정 / 도우미 / 음성
- 홈, 팀원, 설정은 보조 영역으로 내리는 것이 적절함

## 4. 내일까지의 MVP 기능 정의

### 4.1 Must Have
#### A. 앱 셸 동작
- Expo Go에서 실행 가능
- 탭 구조 진입 가능
- 각 탭이 정상적으로 웹 경로를 불러옴

#### B. 핵심 탭
- 홈
- 채터
- 일정
- 문서
- 도우미
- 음성

#### C. Web Harness 안정성
- 로딩 UI 표시
- 오류 시 재시도 가능
- 외부 링크는 외부 브라우저로 열기
- 내부 링크는 WebView 내에서 유지

#### D. 모바일 기본 UX
- Android 뒤로가기 대응
- 현재 화면 새로고침
- 최소 헤더/타이틀 유지

#### E. 앱용 그룹웨어 차별화
- 문서 탭은 작성보다 승인/반려 중심으로 동작
- 앱 메뉴는 웹 사이드바가 아니라 모바일 카드형 메뉴로 구성
- 탭/메뉴 우선순위가 웹과 다르게 적용됨
- 모바일에서 불필요한 작성/설정 CTA는 축소 또는 제거

### 4.2 Should Have
- 홈 화면을 단순 `/`가 아니라 “모바일용 진입 허브”처럼 다듬기
- 브리지 최소 이벤트
  - `OPEN_EXTERNAL_URL`
  - `GET_DEVICE_INFO`
- 앱에서 웹으로 콜백 이벤트 전달

### 4.3 Nice to Have
- 홈에서 빠른 액션 버튼
- 딥링크 기본 처리
- 탭 배지/읽지 않음 표시

## 5. 현재 상태 기준 출발점

현재 이미 확보된 상태:
- `apps/mobile` Expo 기반 구조 생성 완료
- Expo Go 54 계열 호환 확인 완료
- WebView 기반 탭 연결 완료
- 로컬 웹 서버를 앱에서 불러오는 구조 확인 완료
- 외부 링크/기초 브리지 구조 일부 반영 완료

즉 이번 계획은 **0에서 시작하는 것이 아니라, 현재 스캐폴드 위에 MVP 완성도를 높이는 작업**이다.

## 6. 내일까지의 실행 순서

### Phase 1. Web Harness 안정화
목표: “앱에서 웹이 열리는 수준”을 “실사용 가능한 수준”으로 올린다.

작업:
- Android 뒤로가기 처리
- 새로고침 액션 추가
- 로딩/에러/재시도 UX 보강
- 외부 링크 정책 점검
- WebView 공통 옵션 안정화

완료 기준:
- 사용자가 탭 이동, 뒤로가기, 재시도 정도는 자연스럽게 수행 가능

---

### Phase 2. 모바일 탭 우선순위 정리
목표: 앱에서 먼저 봐야 할 기능이 앞에 오도록 정리한다.

권장 탭 우선순위:
1. 홈
2. 채터
3. 일정
4. 문서
5. 도우미
6. 음성

검토 포인트:
- 탭 명칭이 모바일 사용자 기준으로 직관적인가
- 첫 진입이 실제 업무 흐름과 맞는가

완료 기준:
- 앱 첫 사용자가 탭 구조만 보고 주요 기능을 이해할 수 있음

---

### Phase 2-1. 앱용 메뉴/정보구조 재정리
목표: 웹의 좌측 네비게이션을 그대로 줄이지 않고, 앱용 메뉴 흐름으로 바꾼다.

작업:
- 프로필 + 핵심 기능 카드형 메뉴 적용
- 문서/채터/일정/도우미/음성을 메인 영역에 배치
- 홈/팀원/설정은 보조 영역으로 분리
- 메뉴 클릭 시 앱 탭 이동 흐름 보장

완료 기준:
- 메뉴를 열었을 때 웹 사이드바가 아니라 앱 메뉴처럼 느껴짐
- 주요 업무 화면으로 즉시 이동 가능

---

### Phase 3. 홈 화면 최소 허브화
목표: 홈을 단순 웹 `/` 노출이 아니라 “오늘의 업무 진입점”으로 보완할지 결정한다.

선택지:
1. **빠른 일정상 그대로 웹 홈 유지**
2. 앱 네이티브 홈을 얇게 만들고 주요 탭 바로가기 제공

내일까지 기준 권장:
- 일정이 촉박하므로 우선은 웹 홈 유지
- 다만 헤더나 안내 문구로 앱 진입 구조를 보완

완료 기준:
- 홈이 앱 첫 진입점으로서 혼란을 주지 않음

---

### Phase 4. 브리지 최소 기능 고정
목표: 이후 확장을 위해 웹↔앱 인터페이스를 최소한으로 고정한다.

이번 범위:
- `OPEN_EXTERNAL_URL`
- `GET_DEVICE_INFO`
- 오류 응답 규격

완료 기준:
- 웹에서 모바일 기능 호출의 최소 경로가 동작함

---

### Phase 5. 배포/시연 준비
목표: 내일까지 실제 시연 가능한 상태를 만든다.

작업:
- 로컬 실행 가이드 정리
- 모바일 `.env.local` 값 정리
- 웹/앱 동시 실행 체크리스트 작성
- 시연용 대표 시나리오 선정

완료 기준:
- 다른 사람도 정해진 순서로 앱을 띄워 주요 기능을 확인 가능

## 10. 리스크와 대응

### 리스크 1. 실기기 네트워크 이슈
대응:
- `EXPO_PUBLIC_WEB_BASE_URL`을 로컬 PC IP 기준으로 고정
- 웹 서버는 `0.0.0.0` 바인딩
- Expo는 `--lan` 우선, 불가 시 `--tunnel`

### 리스크 2. WebView 내 웹 UX가 모바일에서 답답함
대응:
- 이번에는 네이티브 재작업 대신
  - 뒤로가기
  - 새로고침
  - 외부 링크 처리
  - 로딩 UX
  - 앱 메뉴/탭 우선순위 재정리
  를 먼저 보완

### 리스크 3. 일정 내 구현 범위 과다
대응:
- Must/Should/Nice를 엄격히 분리
- 오늘 안에 Must Have를 먼저 닫기

### 리스크 4. 인증/세션 이슈
대응:
- 내일까지는 Web-owned Auth 유지
- 인증 구조 개편은 하지 않음
