# WorkPresso 모바일 앱 하네스 도입 발표자료

## 문서 목적
본 문서는 WorkPresso 모바일 앱을 기획하고 구성하는 과정에서 **왜 하네스(Harness) 방식을 도입했는지**, **현재 어떤 구조로 구현했는지**, **앞으로 어떻게 확장할 것인지**를 약 5분 내외로 설명하기 위한 발표자료입니다.

## 발표 한 줄 요약
**WorkPresso 모바일 앱은 기존 웹 자산을 최대한 재사용하기 위해, Expo 기반 네이티브 앱 셸 위에 WebView 하네스를 얹는 하이브리드 구조로 설계했습니다.**

---

# 1. 발표 개요

## 1-1. 핵심 메시지
이번 모바일 앱은 처음부터 모든 화면을 네이티브로 다시 만든 것이 아니라,
**기존 웹 서비스를 빠르게 모바일 앱으로 전환하면서도 향후 네이티브 확장을 가능하게 하는 하네스 전략**으로 접근했습니다.

## 1-2. 왜 중요한가
WorkPresso는 이미 웹 중심으로 다음 기능이 구현되어 있습니다.
- 문서
- 일정
- 채터
- 업무 도우미
- 음성/회의 기록

이 상태에서 모바일 앱을 별도로 처음부터 만들면:
- 개발 비용이 커지고
- 기능 누락 위험이 생기고
- 웹/앱 이원화가 빨라집니다.

그래서 저희는 **앱은 Shell, 기능은 Web 재사용, 필요한 부분만 점진적 Native 전환**이라는 전략을 선택했습니다.

---

# 2. 하네스 도입 배경

## 2-1. 현재 프로젝트 특성
기존 구조는 `apps/web` 중심이며, 비즈니스 기능과 API가 이미 풍부하게 구현되어 있습니다.

특히 다음과 같은 장점이 있었습니다.
- `apps/web/src/features/*` 기준 기능 분리
- `apps/web/src/app/api/*` 기준 API 풍부
- 인증, 문서, 일정, 채팅, AI 기능이 이미 동작 중

반면 모바일에서 바로 네이티브로 옮기기 어려운 요소도 있었습니다.
- 웹 전용 UI 의존성
- FullCalendar 같은 웹 중심 컴포넌트
- 기존 인증 흐름이 웹 세션 중심

## 2-2. 결론
이 프로젝트는 Full Native 재작성보다,
**기존 웹을 모바일에서 안전하게 감싸는 Harness 방식이 더 적합**했습니다.

---

# 3. 하네스란 무엇인가

## 3-1. 정의
WorkPresso 모바일 하네스는 크게 두 층으로 나뉩니다.

1. **App Shell (Native Layer)**
   - Expo / React Native
   - 탭 구성
   - 딥링크 처리
   - 뒤로가기/새로고침
   - 외부 링크 처리
   - 브리지 처리

2. **Web Harness (Embedded Web Layer)**
   - WebView 안에서 기존 `apps/web` 화면 로드
   - 기존 Next.js UI와 API 재사용

즉, 앱은 단순 브라우저가 아니라,
**네이티브 UX를 제공하는 앱 셸 + 웹 기능 실행 레이어**로 나뉘어 있습니다.

## 3-2. 핵심 구조
흐름은 아래와 같습니다.

```text
사용자
  → 모바일 앱 탭 진입
    → Native Shell 화면 진입
      → WebView가 대응 웹 URL 로드
        → 기존 WorkPresso 웹 기능 실행
```

그리고 필요할 때는:

```text
웹 화면
  → 브리지 이벤트 전송
    → Native Layer 처리
      → 결과를 다시 웹으로 전달
```

---

# 4. 현재 구현 구조

## 4-1. 앱 셸
관련 파일:
- `apps/mobile/app/_layout.tsx`
- `apps/mobile/app/(tabs)/_layout.tsx`

현재 앱은 Expo Router 기반으로 구성되어 있고,
하단 탭은 다음처럼 구성되어 있습니다.
- 홈
- 채터
- 일정
- 문서
- 도우미
- 음성

즉 모바일 사용자는 일반 앱처럼 탭 UI를 사용하지만,
각 탭 내부의 실제 업무 기능은 WebView를 통해 웹과 연결됩니다.

## 4-2. 화면별 URL 매핑
관련 파일:
- `apps/mobile/src/shared/app-sections.ts`

각 탭은 단순히 웹 홈으로 가는 것이 아니라,
모바일 목적에 맞게 최적화된 URL로 연결됩니다.

예를 들면:
- 문서 탭 → `/documents?mobile=1&scope=approvals&status=PENDING`
- 일정 탭 → `/schedules?mobile=1&focus=today`
- 도우미 탭 → `/chat?mobile=1&context=work`
- 음성 탭 → `/voice?mobile=1&entry=quick-record`

즉 하네스는 웹을 그냥 띄우는 것이 아니라,
**모바일 사용 시나리오에 맞는 진입점으로 웹을 감싸는 구조**입니다.

## 4-3. 공통 WebView 컨테이너
관련 파일:
- `apps/mobile/src/harness/webview/WebViewContainer.tsx`
- `apps/mobile/src/harness/webview/WebScreen.tsx`

이 레이어에서 처리하는 핵심 기능은 다음과 같습니다.
- WebView 로딩
- 에러 처리 및 재시도
- 현재 URL 추적
- 뒤로가기 제어
- 안드로이드 하드웨어 뒤로가기 대응
- 외부 링크 분기 처리
- 내부 링크 탭 전환 처리
- 개발용 브리지 로그 패널 제공

즉 WebView 하나만 띄운 것이 아니라,
**모바일 사용성을 보강하는 공통 실행 하네스**를 별도로 만든 것입니다.

---

# 5. 브리지 도입 이유와 현재 범위

## 5-1. 왜 브리지가 필요한가
웹은 기본적으로 앱의 네이티브 기능을 직접 제어할 수 없습니다.
예를 들면:
- 외부 브라우저 열기
- 기기 정보 확인
- 향후 마이크/파일/공유 처리

그래서 웹과 네이티브 사이에 브리지가 필요합니다.

## 5-2. 현재 구현 범위
관련 파일:
- `apps/mobile/src/harness/bridge/bridge-handler.ts`
- `apps/mobile/src/harness/bridge/injected-bridge.ts`

현재 브리지는 다음 역할을 담당합니다.
- `OPEN_EXTERNAL_URL`
- `GET_DEVICE_INFO`
- 웹 세션 상태 감지
- 웹 라우트 변경 감지
- 웹 런타임 오류 감지
- 브리지 준비 상태 전파

즉 지금 단계에서 브리지는 모든 네이티브 기능을 다 붙인 것이 아니라,
**하네스가 안정적으로 동작하기 위한 최소 핵심 기능부터 구성**했습니다.

## 5-3. 발표 포인트
이 부분은 발표에서 이렇게 설명하면 됩니다.

> 브리지는 웹 기능을 앱 안에 넣기 위한 단순 연결선이 아니라, 웹이 할 수 없는 모바일 고유 기능을 Native Layer가 대신 처리하기 위한 확장 포인트입니다.

---

# 6. 딥링크와 라우팅 처리

## 6-1. 왜 필요했는가
앱으로 들어오는 URL이나 내부 이동을 웹 기준으로만 처리하면,
탭 구조와 앱 네비게이션이 깨질 수 있습니다.

## 6-2. 현재 방식
관련 파일:
- `apps/mobile/src/harness/deeplink/deep-linking.ts`
- `apps/mobile/app/_layout.tsx`

현재 구조는 다음처럼 동작합니다.
- 앱 스킴 `workpresso://...` 또는 내부 웹 URL 수신
- 해당 경로가 탭으로 매핑 가능하면 바로 탭 이동
- 탭으로 매핑되지 않으면 `/web` 화면으로 라우팅

즉 하네스는 단순 WebView 앱이 아니라,
**앱의 네비게이션 구조와 웹 URL 구조를 연결하는 라우팅 레이어**도 함께 가지고 있습니다.

---

# 7. 하네스 도입 효과

## 7-1. 장점
1. **출시 속도가 빠릅니다.**
   - 기존 웹 기능을 즉시 활용 가능

2. **기능 누락 위험이 적습니다.**
   - 이미 검증된 웹 기능을 모바일에서 재사용

3. **소스 오브 트루스가 유지됩니다.**
   - 웹과 앱이 완전히 갈라지지 않음

4. **점진적 네이티브 전환이 가능합니다.**
   - 음성, 채터, 파일 처리 같은 기능부터 순차적으로 Native 전환 가능

## 7-2. 리스크도 존재
물론 하네스 방식에도 한계는 있습니다.
- WebView UX가 완전한 네이티브만큼 부드럽지는 않을 수 있음
- 인증/세션 이슈를 추가로 다뤄야 함
- 마이크/파일/백그라운드 처리에서 웹 한계가 있음

그래서 현재 전략은 명확합니다.

**먼저 하네스로 빠르게 모바일 MVP를 만들고,
모바일 효용이 큰 기능만 네이티브로 단계적으로 추출한다.**

---

# 8. 앞으로의 확장 방향

현재 하네스는 모바일 MVP의 실행 기반을 만드는 단계입니다.
향후 확장 방향은 다음과 같습니다.

1. **인증 안정화**
   - 초기에는 웹 세션 유지 중심
   - 이후 필요 시 Native Auth + Session Bridge 검토

2. **브리지 확장**
   - 마이크 권한
   - 파일 선택
   - 공유 시트
   - 푸시 알림
   - 햅틱 피드백

3. **선별적 네이티브 전환**
   - Pod-D 음성
   - Pod-E 채터
   - 모바일 전용 홈 허브

즉 하네스는 임시방편이 아니라,
**초기 속도와 장기 확장성을 함께 확보하기 위한 중간 아키텍처**입니다.

# 9. 5분 발표 대본
안녕하세요. 오늘은 WorkPresso 모바일 앱에 도입한 하네스 구조에 대해 설명드리겠습니다.

이번 모바일 앱의 핵심은, 모든 기능을 처음부터 네이티브로 다시 만든 것이 아니라, 기존 웹 자산을 최대한 재사용하면서 모바일 앱 형태로 빠르게 전환했다는 점입니다. 저희는 이를 위해 Expo 기반의 네이티브 앱 셸 위에 WebView 하네스를 올리는 전략을 선택했습니다.

먼저 왜 이런 방식을 선택했는지 말씀드리겠습니다. 현재 WorkPresso는 이미 웹 중심으로 문서, 일정, 채터, 업무 도우미, 음성 기능이 구현되어 있고, API도 풍부하게 갖춰져 있습니다. 이런 상황에서 모바일 앱을 별도로 처음부터 네이티브로 다시 구현하면 개발 비용이 커지고, 웹과 앱 간 기능 차이가 생기며, 유지보수 부담도 빠르게 증가합니다. 그래서 저희는 기존 웹의 강점을 유지하면서 모바일 앱 셸만 별도로 구성하는 하네스 방식을 택했습니다.

하네스 구조는 두 층으로 이해하시면 됩니다. 첫 번째는 App Shell, 즉 네이티브 레이어입니다. 여기서는 탭 UI, 딥링크, 뒤로가기, 새로고침, 외부 링크 처리 같은 모바일 UX를 담당합니다. 두 번째는 Web Harness, 즉 WebView 안에서 기존 WorkPresso 웹 화면을 실행하는 레이어입니다. 다시 말해 앱은 단순 브라우저가 아니라, 네이티브 UX를 제공하는 껍데기와 실제 업무 기능을 실행하는 웹 레이어가 결합된 구조입니다.

현재 구현을 보면, 모바일 앱은 Expo Router 기반으로 홈, 채터, 일정, 문서, 도우미, 음성 탭을 가지고 있습니다. 그리고 각 탭은 단순히 웹 홈으로 가는 것이 아니라, 모바일 목적에 맞는 URL로 바로 연결됩니다. 예를 들어 문서 탭은 승인 대기 문서 중심으로 진입하고, 일정 탭은 오늘 일정 중심으로 진입하며, 도우미 탭은 모바일 업무 질의에 맞는 컨텍스트로 들어갑니다. 즉 단순한 웹 포장 수준이 아니라, 모바일 사용자 흐름에 맞게 진입점을 설계한 것입니다.

또 하나 중요한 부분은 공통 WebView 하네스입니다. 이 레이어에서는 WebView 로딩과 에러 처리, 재시도, 현재 URL 추적, 안드로이드 뒤로가기 대응, 외부 링크 분기, 내부 링크의 탭 이동 처리까지 담당합니다. 개발 단계에서는 브리지 로그 패널도 제공해서 웹과 앱 사이 메시지를 확인할 수 있도록 했습니다. 즉, 그냥 WebView를 띄운 것이 아니라 실제 모바일 사용성을 확보하기 위한 실행 환경을 따로 구축한 것입니다.

브리지도 중요한 요소입니다. 웹은 기본적으로 앱의 네이티브 기능을 직접 호출할 수 없기 때문에, 외부 링크 열기나 기기 정보 조회 같은 작업은 Native Layer가 대신 처리해야 합니다. 현재는 브리지를 통해 외부 URL 열기, 기기 정보 제공, 웹 세션 상태 감지, 웹 라우트 변경 감지, 런타임 오류 감지 등을 지원하고 있습니다. 이 브리지는 앞으로 마이크, 파일 선택, 공유 시트, 푸시 알림 같은 모바일 고유 기능으로 확장될 수 있는 기반입니다.

딥링크 처리도 같이 설계했습니다. 앱 스킴이나 내부 웹 URL이 들어오면, 해당 경로가 탭으로 연결 가능한 경우에는 바로 탭으로 보내고, 그렇지 않은 경우에는 별도의 웹 화면으로 라우팅합니다. 덕분에 앱의 네비게이션 구조와 웹 URL 구조를 함께 유지할 수 있습니다.

정리하면, 하네스 도입의 장점은 세 가지입니다. 첫째, 출시 속도가 빠릅니다. 둘째, 이미 검증된 웹 기능을 활용하기 때문에 기능 누락 위험이 낮습니다. 셋째, 웹과 앱의 소스 오브 트루스를 불필요하게 나누지 않으면서도, 나중에 필요한 기능만 네이티브로 점진적으로 전환할 수 있습니다.

결국 WorkPresso 모바일 앱의 하네스는 단순한 임시 WebView 전략이 아니라, 빠른 모바일 전환과 장기적인 네이티브 확장을 동시에 고려한 현실적인 아키텍처라고 말씀드릴 수 있습니다. 감사합니다.

# 10. 발표 마무리 한 줄
**WorkPresso 모바일 하네스는 웹의 완성도를 유지하면서, 앱 셸과 브리지로 모바일 확장성을 확보한 전략적 중간 아키텍처입니다.**
