# Pod-C 발표 정리 및 5분 발표 대본

## 1. 발표 목적
이 발표는 WorkPresso의 Pod-C가 어떤 역할을 하는지, RAG를 어떤 범위에서 사용했는지, 그리고 실제로 회사 내부 지식을 어떤 방식으로 불러오는지를 설명하기 위한 자료입니다.

---

## 2. 한 줄 소개
**Pod-C는 사내 문서와 업무 데이터를 연결해, 정적 내부 지식은 RAG로 검색하고 실시간 업무 정보는 툴 호출로 처리하는 업무형 AI 비서입니다.**

---

## 3. Pod-C의 역할
Pod-C는 단순한 자유대화형 챗봇이 아니라, 다음 두 가지 역할에 집중하도록 한정적으로 설계되었습니다.

1. **사내 지식 검색형 응답**
   - 개발 표준
   - 복리후생
   - 보안 수칙
   - 근무 환경
   - 운영 가이드
   - 회의록, 일정, 캡처된 스레드 기반 정보

2. **필요한 경우 실제 업무 툴 호출**
   - 일정 생성
   - 결재 대기 문서 목록 조회

즉, Pod-C는 "모든 것을 아는 AI"가 아니라, **회사 내부 맥락 안에서 정확하게 응답하도록 제한된 AI**입니다.

---

## 4. Pod-C의 핵심 구조
핵심 채팅 진입점은 아래 파일에 있습니다.

- `apps/web/src/app/api/chat/route.ts`

전체 흐름은 다음과 같습니다.

1. 사용자가 질문을 입력한다.
2. Pod-C는 마지막 사용자 질문을 기준으로 임베딩을 만든다.
3. 벡터 저장소에서 관련 내부 지식을 검색한다.
4. 검색된 내용을 시스템 프롬프트에 넣는다.
5. LLM이 답변을 생성한다.
6. 필요한 경우 일정 생성이나 문서 조회 같은 tool call을 실행한다.

정리하면 다음과 같습니다.

**사용자 질문 → 임베딩 생성 → 벡터 검색 → 검색 결과를 프롬프트에 주입 → LLM 응답 생성 → 필요 시 툴 호출**

---

## 5. RAG는 어떻게 사용되었는가

### 5-1. 저장 대상
Pod-C는 여러 종류의 내부 데이터를 벡터화할 수 있습니다.

관련 정의 파일:
- `apps/web/src/features/pod-c/services/knowledge-contract.ts`

지식 소스 타입:
- `DOCUMENTS`
- `MEETING_LOGS`
- `SCHEDULES`
- `CHAT_THREADS`

즉 Pod-C는 단순 문서 검색을 넘어서,
**문서, 회의록, 일정, 메신저 스레드까지 통합적으로 검색 가능한 구조**입니다.

### 5-2. 벡터 저장 방식
관련 파일:
- `apps/web/src/features/pod-c/services/knowledge-sync.ts`
- `packages/db/schema.db`

벡터 저장소 테이블은 `knowledge_vectors`이며, 주요 정보는 다음과 같습니다.

- `source_type`: 데이터 출처 타입
- `source_id`: 원본 데이터 ID
- `embedding`: 임베딩 벡터
- `metadata`: 제목, 본문, 부가 정보

즉 Pod-C는 원문 텍스트만 저장하는 것이 아니라,
**출처와 메타데이터를 함께 저장해 어떤 지식이 어디서 왔는지 추적 가능하게 설계**했습니다.

### 5-3. 임베딩 생성 방식
관련 파일:
- `apps/web/src/lib/ai/embeddings.ts`
- `apps/web/src/features/pod-c/services/knowledge-sync.ts`

처리 방식:
- 제목과 본문을 합쳐 임베딩 입력을 구성
- OpenRouter 기반 임베딩 모델로 벡터 생성
- `knowledge_vectors`에 upsert

즉, Pod-C는 문서를 그대로 읽는 게 아니라,
**검색 가능한 벡터 지식으로 변환해두고 필요할 때 유사도 검색을 수행**합니다.

### 5-4. 검색 방식
관련 파일:
- `packages/db/schema.db`
- `apps/web/src/app/api/chat/route.ts`

검색은 `match_knowledge` 함수로 수행됩니다.

검색 로직:
- 질문 임베딩 생성
- `match_knowledge(query_embedding, match_threshold, match_count)` 호출
- 유사도가 높은 상위 문서 검색
- 검색된 `metadata.content`를 프롬프트에 삽입
- 해당 문맥을 기반으로 답변 생성

즉, Pod-C는 **먼저 사내 문서를 찾고, 그 문서를 읽은 뒤 답하는 형태의 RAG 구조**입니다.

---

## 6. 회사 내부 내용은 어떻게 들어오는가

### 6-1. 공식 가이드 문서 기반
관련 파일:
- `apps/web/src/features/pod-c/knowledge-base/internal-guide.md`
- `apps/web/src/app/api/admin/sync-knowledge/route.ts`

예를 들어 `internal-guide.md`에는 다음과 같은 회사 내부 정보가 들어 있습니다.

- 개발 표준
- 복리후생
- 예외 및 주의사항
- 근무 환경
- 보안 수칙

이 문서들은 관리자 동기화 API를 통해 Pod-C의 지식 저장소로 들어갑니다.

흐름은 다음과 같습니다.

**사내 Markdown 가이드 → sync API → 임베딩 생성 → knowledge_vectors 저장 → Pod-C 검색 가능**

### 6-2. 업무 데이터 기반
사내 지식은 정적인 문서만이 아니라 실제 업무 데이터에서도 들어옵니다.

- 일정 생성 시 → `SCHEDULES`로 동기화
  - 관련 파일: `apps/web/src/app/api/schedules/route.ts`
- 회의록 정제 시 → `MEETING_LOGS`로 동기화
  - 관련 파일: `apps/web/src/app/api/internal/pod-d/meeting-refined/route.ts`
- 문서 승인 상태 변화 시 → `DOCUMENTS`로 동기화
  - 관련 파일: `apps/web/src/features/pod-a/services/document-server.ts`
- 메신저 스레드 캡처 시 → `CHAT_THREADS`로 동기화
  - 관련 파일: `apps/web/src/features/pod-e/services/chatter-service.ts`

즉 Pod-C는 고정된 문서만 보는 것이 아니라,
**회사에서 실제로 생성되는 업무 산출물도 지식으로 축적하는 구조**입니다.

## 7. 실제 사용자가 어떻게 회사 내용을 불러오는가
관련 파일:
- `apps/web/src/features/pod-c/components/chat-panel.tsx`

Pod-C에는 빠른 명령이 들어가 있습니다.

- `/개발표준`
- `/복리후생`
- `/주의사항`
- `/근무환경`
- `/보안수칙`

사용자는 슬래시 명령을 누르거나 자유롭게 질문할 수 있습니다.

예시:
- “우리 회사 코어 타임이 뭐야?”
- “복리후생 알려줘”
- “보안 수칙 정리해줘”
- “개발 표준이 어떻게 돼?”

이 요청이 들어오면 Pod-C는:
1. 질문 임베딩 생성
2. 관련 내부 문서 검색
3. 검색 결과를 프롬프트에 포함
4. 자연어로 재구성해 응답

즉, **외부 일반 지식이 아니라 회사 내부 문서를 근거로 대답한다는 점**이 핵심입니다.

---

## 8. 왜 모든 것을 RAG로만 처리하지 않았는가
Pod-C는 RAG만으로 동작하지 않습니다.

이유는 실시간성이 필요한 데이터는 RAG만으로 처리하면 정확도가 떨어질 수 있기 때문입니다.

대표 사례:
- “현재 결재 대기 문서 보여줘”
- “내가 승인해야 할 문서 있어?”

이런 요청은 과거에 저장된 지식이 아니라 **지금 시점의 상태값**이 중요합니다.
그래서 현재 구조에서는 문서 목록 조회 툴을 별도로 둬서 처리합니다.

즉 역할 분리는 다음과 같습니다.

- **정적/반정적 내부 지식** → RAG
- **실시간 상태 조회 및 액션** → Tool Calling

이 설계 덕분에 설명형 질문과 실행형 질문을 분리해서 더 안정적으로 처리할 수 있습니다.

---

## 9. 발표에서 강조할 포인트

### 포인트 1. 업무형 AI라는 점
Pod-C는 범용 챗봇이 아니라, 회사 내부 맥락에서 동작하도록 제한된 AI입니다.

### 포인트 2. RAG와 Tool Calling의 역할 분리
정적 지식은 RAG로, 실시간 업무 데이터는 툴로 처리합니다.

### 포인트 3. 데이터 소스가 다양하다는 점
문서뿐 아니라 일정, 회의록, 채팅 스레드까지 지식화할 수 있습니다.

### 포인트 4. 실제 업무 흐름과 연결된 구조
단순 답변에 그치지 않고, 일정 생성이나 결재 문서 조회처럼 실제 업무와 연결됩니다.

## 10. 5분 발표 대본
안녕하세요. 제가 소개드릴 부분은 WorkPresso의 Pod-C, 즉 사내 업무 비서 AI입니다.

Pod-C를 한 문장으로 설명하면, 회사 내부 지식과 업무 데이터를 연결해서 필요한 정보를 찾아주고, 경우에 따라 실제 업무 기능까지 연결해주는 AI 컴포넌트입니다.

먼저 중요한 점은 Pod-C가 범용 챗봇이 아니라는 점입니다. 저희는 Pod-C를 회사 안에서 필요한 질문에 더 정확하게 답하도록 한정적으로 설계했습니다. 예를 들어 개발 표준, 복리후생, 보안 수칙, 근무 환경 같은 내부 운영 정보에 답할 수 있고, 회의록이나 일정, 메신저 스레드 같은 업무 데이터도 검색 대상으로 활용할 수 있습니다.

Pod-C의 핵심 동작 방식은 RAG입니다. 사용자가 질문을 입력하면 Pod-C는 먼저 그 질문을 임베딩으로 변환합니다. 그리고 벡터 저장소에서 질문과 의미적으로 가까운 사내 지식들을 검색합니다. 여기서 검색된 문서 내용이 다시 시스템 프롬프트 안으로 들어가고, 그 문맥을 기반으로 최종 답변을 생성합니다. 즉 모델이 그냥 아는 척해서 답하는 것이 아니라, 먼저 회사 내부 문서를 찾고 그 문서를 읽은 뒤 답하도록 만든 구조입니다.

이때 저장되는 지식은 단순 텍스트가 아닙니다. 문서, 회의록, 일정, 채팅 스레드 등 여러 데이터를 source type과 source id, metadata와 함께 벡터 테이블에 저장합니다. 그래서 나중에 어떤 답변이 어떤 종류의 데이터에서 왔는지 추적하기 쉽고, 검색 대상도 문서 하나에만 묶이지 않습니다.

실제 회사 내부 정보는 어떻게 들어오느냐를 말씀드리면, 대표적으로 Pod-C의 knowledge-base 폴더에 있는 공식 운영 가이드 문서가 있습니다. 여기에는 개발 표준, 복리후생, 주의사항, 근무 환경, 보안 수칙 등이 정리되어 있습니다. 이 문서는 동기화 API를 통해 벡터 저장소에 들어가고, 이후 사용자가 “우리 회사 코어 타임이 뭐야?” 혹은 “복리후생 알려줘” 같은 질문을 하면 Pod-C가 이 내부 문서를 검색해서 답변하게 됩니다.

또한 Pod-C는 정적인 가이드 문서만 보는 것이 아닙니다. 일정이 생성되면 일정 정보가 지식으로 들어가고, 회의록이 정리되면 회의 요약과 액션 아이템이 지식화됩니다. 메신저에서 중요한 스레드를 캡처하면 그것도 나중에 검색 가능한 지식으로 축적됩니다. 즉 Pod-C는 단순 FAQ 챗봇이 아니라, 실제 업무 과정에서 생성되는 데이터가 계속 쌓이는 지식 허브에 가깝습니다.

다만 저희는 모든 것을 RAG로 해결하지는 않았습니다. 예를 들어 “현재 결재 대기 중인 문서를 보여줘” 같은 요청은 실시간 상태가 중요합니다. 이런 경우에는 벡터 검색만으로 처리하면 시점 차이 때문에 부정확할 수 있습니다. 그래서 이 부분은 문서 조회 툴을 따로 연결해서 현재 상태를 직접 조회하도록 설계했습니다. 마찬가지로 일정 생성도 단순 답변이 아니라 별도 tool call을 통해 실제 API를 호출하도록 분리했습니다.

정리하면, Pod-C의 핵심은 두 가지입니다. 첫째, 정적이거나 반정적인 회사 내부 지식은 RAG로 검색해서 설명형 답변을 제공한다는 점입니다. 둘째, 현재 상태 확인이나 실제 실행이 필요한 요청은 툴 호출로 분리해서 정확성을 확보했다는 점입니다.

그래서 Pod-C는 단순한 채팅 AI가 아니라, 회사 내부 문맥을 이해하고 실제 업무 흐름과 연결되는 실용적인 AI 비서라고 말씀드릴 수 있습니다. 감사합니다.

## 11. 발표 마무리용 한 줄
**Pod-C는 사내 지식은 RAG로, 실시간 업무는 Tool Calling으로 처리하는 하이브리드형 업무 비서입니다.**
