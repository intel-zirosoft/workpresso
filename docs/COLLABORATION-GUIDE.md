# WorkPresso Collaboration & AI Guide
<!-- WorkPresso 협업 및 AI 활용 가이드 -->

이 문서는 WorkPresso 프로젝트의 팀원들이 `SKILL.md` 파일을 어떻게 이해하고 활용해야 하는지, 그리고 우리의 협업 방식이 어떤 가치를 지향하는지 설명합니다.

---

## 1. 왜 Pod별로 SKILL.md를 나누었나요? (Domain Isolation)
AI 에이전트(Gemini, Cursor 등)는 한 번에 처리할 수 있는 정보량(Context Window)에 한계가 있습니다. 모든 정보를 한곳에 몰아넣으면 AI가 핵심 규칙을 놓치거나 엉뚱한 코드를 생성할 확률이 높아집니다.
- **AI 최적화**: 각 Pod 담당 에이전트가 본인의 도메인 규칙에만 집중하게 하여 정확도를 극대화합니다.
- **도메인 책임**: 각 팀원이 본인 도메인의 '전문가 에이전트'를 소유하고 관리하는 개념입니다.

## 2. 풀스택 수직적 분할 (Full-stack Vertical Slicing)
우리는 프론트엔드와 백엔드를 나누지 않습니다. 대신 **'기능(Feature)'**을 나눕니다.
- **병목 제거**: "백엔드 API 나올 때까지 기다릴게요"라는 상황을 없앱니다. 한 명이 DB 설계부터 UI 구현까지 한 흐름을 완성합니다.
- **성장 경험**: 모든 팀원이 데이터의 흐름(DB -> API -> UI) 전체를 조망하는 풀스택 개발 역량을 갖춥니다.
- **2인 페어 협업**: 기능을 더 작은 단위로 쪼개어 각자 풀스택으로 개발하되, 서로의 코드를 리뷰하며 시스템 전체를 이해합니다.

## 3. AI Protocol: 바이브 코딩의 안전장치
AI와 함께하는 '바이브 코딩'은 빠르지만 위험할 수 있습니다. 이를 방지하기 위한 규약입니다.
- **schema.db 참조**: AI는 실제 DB 구조를 모른 채 코드를 짤 때가 많습니다. 반드시 `schema.db`를 읽게 하여 필드명 오타나 타입 에러를 방지하세요.
- **Zod Schema First**: 클라이언트와 서버가 공유할 데이터 규격(Zod)을 먼저 정의하면, AI가 프론트와 백엔드 코드를 짤 때 서로 어긋나지 않습니다.

## 4. 의존성 및 스키마 관리 (Source of Truth)
- **DEPENDENCIES.md**: 새로운 라이브러리 설치 시 에이전트가 자동으로 기록하게 합니다. 이는 팀원 간의 싱크를 맞추고 중복 설치를 막는 소통 창구가 됩니다.
- **schema.db**: 우리 프로젝트의 유일한 진실의 원천(Source of Truth)입니다. 코딩 전 항상 이 파일을 확인하는 습관을 들여주세요.

## 5. 문서 형식 (English + Korean Comments)
- **글로벌 표준**: AI 모델은 영어 지시문을 훨씬 더 정확하게 이해하고 수행합니다.
- **한국어 주석**: 팀원들의 빠른 이해를 돕기 위해 마크다운 주석(`<!-- -->`)으로 설명을 병기했습니다.
